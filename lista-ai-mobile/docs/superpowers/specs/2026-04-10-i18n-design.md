# i18n Design — Lista AI Mobile

**Date:** 2026-04-10
**Status:** Implemented

## Goal

Add full internationalization (i18n) support to the Lista AI React Native app with auto-detection on first launch and a manual language override persisted across sessions.

**Supported locales:** English (en), Portuguese BR (pt-BR), Spanish (es), French (fr), German (de)

---

## Architecture Decision: Zustand owns state, i18next is a service

**Chosen approach:** Zustand holds the active `Locale` string. i18next is initialized synchronously at module load (static JSON imports, no async resource loading) and used purely as a formatting/translation service.

**Rejected approaches:**
- *Zustand + MMKV:* Faster storage but adds a native module dependency without sufficient justification.
- *Expo Localization as source of truth:* Read-only; can't hold a user override.

**Why this split:** Components use `useTranslation()` from react-i18next (auto re-renders on language change). Non-component code (Zustand actions, auth store errors) uses `i18n.t()` directly. The language persists in AsyncStorage under key `app_language`.

---

## Bootstrap Sequence

```
App.tsx renders → import './src/i18n' (synchronous init, lng: 'en') 
               → useEffect: initLanguage()
                   → AsyncStorage.getItem('app_language')
                   → fallback: Localization.getLocales()[0].languageTag
                   → matchLocale() → i18n.changeLanguage() + set Zustand
               → setLangReady(true) → SplashScreen.hideAsync()
```

`initLanguage` is guarded with try/catch so AsyncStorage failures fall back silently to the `'en'` set during synchronous init.

## Locale Matching

```typescript
function matchLocale(tag: string): Locale {
  if (SUPPORTED_LOCALES.includes(tag)) return tag;       // exact: 'pt-BR'
  if (tag.split('-')[0] === 'pt') return 'pt-BR';        // 'pt-PT' → pt-BR
  const match = SUPPORTED_LOCALES.find(l => l.startsWith(tag.split('-')[0]));
  return match ?? 'en';                                   // 'fr-CA' → 'fr'
}
```

## Translation File Structure

Single file per locale under `src/i18n/locales/`:

```
en.json
pt-BR.json
es.json
fr.json
de.json
```

Keys are organized by domain:

| Namespace | Coverage |
|-----------|----------|
| `common`  | Cancel, Save, Delete, Retry, Error, or |
| `auth`    | Shared fields + login/register sub-keys |
| `lists`   | ListsHome, ListDetail, AddEditList, card |
| `items`   | AddEditItem screen |
| `settings` | Settings screen + language names |
| `profile` | Perfil, PerfilInfo screens |
| `navigation` | Bottom tab labels |
| `sync`    | SyncStatusBar (with pluralization) |
| `errorBoundary` | ErrorBoundary fallback |
| `shopping` | Compras placeholder screen |

Pluralization uses i18next convention: `sync.syncing_one` / `sync.syncing_other` resolved via `t('sync.syncing', { count: N })`.

Interpolation: `lists.deleteDialog.message` uses `{{name}}`; `lists.card.percentComplete` uses `{{percent}}`.

## Settings Screen

The Settings screen is the entry point for language switching. Design decision: **bottom sheet picker** (chosen over inline list to keep the Settings screen minimal and extensible as new settings rows are added).

The language row shows the current locale name and a chevron. Tapping opens a `Modal` with `animationType="slide"` containing all 5 locales. The active locale shows a green checkmark.

Sync retry row is **conditionally rendered** (only when `lastSyncError || pendingCount > 0`) to avoid showing a useless button in the normal happy path.

---

## Class Component + Hooks

`ErrorBoundary` is a class component and cannot use `useTranslation()` directly. Solution: extracted a `ErrorFallback` functional component that uses the hook, and the class component's `render()` delegates to it. This is the standard React pattern for this case.

---

## German Register

All German translations use the informal **du** register consistently (initial draft mixed `du` and `Sie`; corrected before merge).
